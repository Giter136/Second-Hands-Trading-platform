# 01. 技术栈选型与本地运行架构说明

更新时间: 2026-04-18

## 0. 项目形态声明

1. 项目形态锁定为网页项目(Web，B/S架构)。
2. 用户通过浏览器访问系统，不提供桌面客户端安装包。
3. 本阶段不开发移动原生客户端与小程序端。
4. 当前阶段定位为小 demo，优先本地快速验证，不引入容器化部署链路。

## 1. 选型目标

1. 轻量可维护: 满足MVP闭环，不引入非必要中间件。
2. AI友好: 使用生态成熟、文档完整、AI生成稳定的主流栈。
3. 易部署: 网页项目在单机本地环境即可启动，降低外部依赖和运维复杂度。

## 3. 选型结论

1. 前后端分离但同仓管理: 前端 Next.js，后端 FastAPI。
2. 数据层仅保留 MySQL，不引入 Redis、MQ 等非MVP必需组件。
3. 文件上传采用后端本地目录持久化(uploads目录)。

## 4. 部署架构(本地单机版)

用户浏览器
-> 本地前端服务(web, 3000)
-> 本地后端服务(api, 8000)
-> 本地数据库服务(mysql, 3306)

说明:

1. 前端服务负责页面渲染和前端资源响应。
2. 后端服务提供业务API，处理鉴权、状态流转与数据读写。
3. MySQL 服务采用本地数据目录持久化，保证重启不丢数据。
4. 服务间通信仅限本机环回地址，不暴露公网访问。

## 5. 本地部署与运行规范

1. 采用本地直跑模式: 前端服务、后端服务、数据库服务分别在本机启动。
2. 环境分层仅保留: local(开发) 与 local-test(联调验证)。
3. 日志策略: 前后端统一输出到本地日志目录，按天滚动。
4. 数据库备份策略: MySQL 定期导出本地备份，上传目录做本地增量备份。
5. 本阶段不要求容器编排、镜像管理和注册表推送流程。

## 6. 非目标(本阶段不做)

1. 公网服务器部署与域名接入。
2. Nginx 反向代理与HTTPS证书自动化。
3. Docker 容器化与多容器编排。
4. Kubernetes集群编排与自动弹性扩缩容。
