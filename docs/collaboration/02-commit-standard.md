# 02. 标准化提交规范

更新时间: 2026-04-18

## 1. Commit Message强制格式

格式: 前缀 + 空格 + 简述

可用前缀:

1. feat 新功能(仅限MVP范围内)
2. fix 缺陷修复
3. docs 文档变更
4. refactor 重构(不改变外部行为)
5. test 测试相关
6. chore 构建、依赖、流程类维护

简述要求:

1. 使用中文或英文短句均可。
2. 必须直接说明变更结果，不写空泛描述。
3. 建议长度 8 到 40 字符。

## 2. 提交颗粒度强约束

1. 一次提交只做一件事。
2. 禁止将不相关改动打包到同一提交。
3. 功能改动与格式化改动必须分离提交。
4. 文档变更与功能变更必须分离提交。
5. 每次提交必须可独立回滚，不依赖后续提交才能成立。

## 3. 典型示例清单

1. feat 商品发布增加必填校验提示
2. fix 交易完成后商品状态未同步问题
3. docs 更新MVP状态机说明
4. refactor 拆分交易服务与数据访问职责
5. test 补充商品发布字段校验用例
6. chore 锁定后端依赖版本

## 4. 禁止示例

1. update stuff
2. fix bug and add feature
3. 一次提交同时包含接口重构、页面改版、文档更新
4. 无说明的临时提交

## 5. 评审最小检查项

1. 提交标题是否符合前缀格式。
2. 提交内容是否单一职责。
3. 是否存在隐藏的跨模块顺手修改。
4. 是否与当前任务目标直接相关。
