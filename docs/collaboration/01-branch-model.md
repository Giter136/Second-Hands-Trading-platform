# 01. 极简分支模型与流转说明

更新时间: 2026-04-18

## 1. 分支模型设计目标

1. 只保留必须分支，减少上下文污染。
2. 适配 AI 高频小步提交场景。
3. 保证主干稳定、问题可快速回滚。

## 2. 分支清单

| 分支类型 | 分支名规则 | 生命周期 | 用途 |
|---|---|---|---|
| 长期分支 | main | 长期存在 | 本地稳定里程碑版本，必须保持可运行 |
| 长期分支 | develop | 长期存在 | 日常集成分支，承接所有已评审改动 |
| 短期分支 | fix/<简述> | 1天到3天 | 日常修复与小范围改进 |
| 短期分支 | hotfix/<简述> | 数小时到1天 | 本地阻塞问题紧急修复，仅在必要时启用 |

## 3. 合并方向(强制)

1. 日常修复: fix/* -> develop。
2. 里程碑发布: develop -> main。
3. 本地阻塞问题紧急修复: hotfix/* -> main，然后 main -> develop 回灌。
4. 禁止反向合并: main 不得直接回合并到 fix/*。

## 4. 流转规则

1. 新任务默认从 develop 切出 fix/*。
2. 短期分支合并后必须删除，禁止长期堆积。
3. main 只接收经过人工评审的合并请求。
4. 任一分支若超出生命周期未完成，必须拆分任务后重新提交。

## 5. 极简发布节奏建议

1. develop 每日可多次合并。
2. main 每周1次本地里程碑发布，紧急问题走 hotfix 通道。
3. 每次发布必须对应清晰的变更说明与回滚点。
