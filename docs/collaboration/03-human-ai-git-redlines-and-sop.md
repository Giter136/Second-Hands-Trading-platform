# 03. 人机协作Git操作红线与异常处理SOP

更新时间: 2026-04-18

## 1. 权限边界总则

1. AI 负责改动生成、变更说明、风险提示。
2. 人类负责仓库控制权操作与最终合并决策。
3. 任何高风险 Git 操作默认由人类执行。

## 2. 人类与AI权限划分

| 操作类别 | AI权限 | 人类权限 | 规则 |
|---|---|---|---|
| 查看状态与差异 | 允许 | 允许 | AI可执行，用于分析上下文 |
| 新建短期分支 | 条件允许 | 允许 | 需基于已确认任务名 |
| 本地提交 | 条件允许 | 允许 | 必须遵守提交规范 |
| 合并分支 | 禁止 | 允许 | 合并由人类主导与确认 |
| 推送远程 | 严禁 | 允许 | AI不得执行推送 |
| 强制覆盖历史 | 严禁 | 严禁(常规) | 非事故应急不得使用 |
| 删除远程分支 | 严禁 | 允许 | 由人类在确认后执行 |
| 打标签与里程碑发布 | 禁止 | 允许 | 发布动作必须人工完成 |

## 3. VibeCoding红线(必须遵守)

1. 严禁让AI执行推送到远程仓库。
2. 严禁让AI执行强制改写历史的操作。
3. 严禁让AI删除远程分支或发布标签。
4. 任何合并到 main 的动作必须由人类发起并确认。
5. AI 发现冲突、失败或异常时必须立即停止继续修改并报告。

## 4. AI改动导致冲突或报错时的标准SOP

### 4.1 触发条件

1. 合并冲突。
2. 构建失败或测试失败。
3. 关键流程回归(本地启动、浏览、沟通、交易任一闭环中断)。

### 4.2 处理流程

1. 立即冻结: 停止新增改动，保持当前现场不继续扩散。
2. 快速定位: 记录异常范围、受影响文件、最近提交。
3. 风险分级:
- P0: 影响主流程或主分支稳定。
- P1: 影响单模块但可绕行。
- P2: 非阻塞性问题。
4. 选择回退策略:
- 策略A: 通过反向提交回退最近错误改动。
- 策略B: 回退到最近稳定检查点后重新拆分任务。
- 策略C: 保留有效改动，剥离有风险改动后重提。
5. 人工复核: 由人类确认回退结果和下一步处理路径。
6. 复盘沉淀: 记录根因、触发条件、预防动作，更新协作文档。

### 4.3 恢复准入条件

1. 主流程恢复可用。
2. 关键报错已消除。
3. 回滚或修复过程有明确记录。
4. 人类确认允许继续迭代后，AI方可继续执行。
